This guide is for teams that want the QTI schema and authoring behavior, but do not want to use the registry web components for toolbar, side panels, or editor chrome.

The core idea is simple:

- keep the ProseMirror or ProseKit editor instance in your own React state
- subscribe to content and selection events
- read and write through the underlying `EditorView`
- register your own ProseMirror plugins alongside the QTI interaction plugins

## When to use this approach

Use this when:

- you already have a React design system
- you want React components for toolbars, inspectors, or sidebars
- you want custom ProseMirror plugins that are specific to your product
- you want QTI authoring behavior without depending on the registry UI

## 1. Keep access to the core editor view

If you use ProseKit as the host, you still have access to the underlying ProseMirror view:

```tsx
import { createEditor, union, type Editor } from 'prosekit/core';

const editor = createEditor({
  extension: union(
    defineBasicExtension(),
    defineQtiInteractionsExtension(),
    qtiEditorEventsExtension({ eventTarget }),
  ),
});

editor.mount(mountElement);

const view = editor.view;
const state = editor.view.state;
```

If you use plain ProseMirror, you already work directly with `EditorView`.

## 2. Bridge editor events into React state

Keep the event bridge local to your app. The event extension emits document and selection snapshots onto an `EventTarget`.

```tsx
// use-qti-editor-state.ts
import { useEffect, useState } from 'react';

export interface EditorSnapshot {
  html: string;
  json: unknown;
  selection: { from: number; to: number; empty: boolean } | null;
}

export function useQtiEditorState(eventTarget: EventTarget) {
  const [snapshot, setSnapshot] = useState<EditorSnapshot>({
    html: '',
    json: null,
    selection: null,
  });

  useEffect(() => {
    const onContent = (event: Event) => {
      const detail = (event as CustomEvent<{ html: string; json: unknown }>).detail;
      setSnapshot(current => ({
        ...current,
        html: detail.html,
        json: detail.json,
      }));
    };

    const onSelection = (event: Event) => {
      const detail = (event as CustomEvent<{ from: number; to: number; empty: boolean }>).detail;
      setSnapshot(current => ({
        ...current,
        selection: detail,
      }));
    };

    eventTarget.addEventListener('qti:content:change', onContent);
    eventTarget.addEventListener('qti:selection:change', onSelection);

    return () => {
      eventTarget.removeEventListener('qti:content:change', onContent);
      eventTarget.removeEventListener('qti:selection:change', onSelection);
    };
  }, [eventTarget]);

  return snapshot;
}
```

That gives your React components enough information to render custom inspectors, preview panes, validation badges, or selection-aware controls.

## 3. Build React-native controls against `EditorView`

Your toolbar or side panel can read from `editor.view.state` and dispatch transactions directly.

```tsx
// QtiSelectionPanel.tsx
import { TextSelection } from 'prosemirror-state';
import type { Editor } from 'prosekit/core';

export function QtiSelectionPanel({ editor }: { editor: Editor | null }) {
  if (!editor?.mounted) return null;

  const view = editor.view;
  const { from, to, empty } = view.state.selection;

  const selectStart = () => {
    const tr = view.state.tr.setSelection(TextSelection.create(view.state.doc, 1));
    view.dispatch(tr);
    view.focus();
  };

  return (
    <aside>
      <p>Selection: {from} - {to}</p>
      <button onClick={selectStart} disabled={empty && from === 1}>
        Jump to start
      </button>
    </aside>
  );
}
```

The same pattern works for custom attribute editors. Read the current node from `view.state.selection`, compute the next attrs, and dispatch a transaction or call an existing QTI command from the relevant interaction package.

## 4. Add your own ProseMirror plugins

You can register product-specific plugins next to the QTI interaction plugins.

```tsx
// custom-word-count-plugin.ts
import { Plugin } from 'prosekit/pm/state';
import { definePlugin } from 'prosekit/core';

export function customWordCountExtension(eventTarget: EventTarget) {
  return definePlugin(
    () =>
      new Plugin({
        view(view) {
          const emit = () => {
            const text = view.state.doc.textContent;
            const words = text.trim() ? text.trim().split(/\s+/).length : 0;

            eventTarget.dispatchEvent(
              new CustomEvent('app:word-count', {
                detail: { words },
                bubbles: true,
              }),
            );
          };

          emit();

          return {
            update(updatedView, prevState) {
              if (!prevState.doc.eq(updatedView.state.doc)) emit();
            },
          };
        },
      }),
  );
}
```

Then include it in your editor assembly:

```tsx
const editor = createEditor({
  extension: union(
    defineBasicExtension(),
    defineQtiInteractionsExtension(),
    qtiEditorEventsExtension({ eventTarget }),
    customWordCountExtension(eventTarget),
  ),
});
```

## 5. Export valid QTI from your own React UI

Once you have access to the document node, export can stay completely separate from the registry components.

```tsx
import { createQtiPackageFromNode } from '@qti-editor/qti-roundtrip-export';

async function exportCurrentItem(editor: Editor, metadata: {
  identifier?: string;
  title?: string;
  lang?: string;
}) {
  const blob = await createQtiPackageFromNode(editor.view.state.doc, {
    identifier: metadata.identifier ?? 'item-1',
    title: metadata.title ?? 'Untitled',
    lang: metadata.lang ?? 'en',
    packageIdentifier: metadata.identifier ?? 'package-1',
    testTitle: metadata.title ?? 'Assessment',
  });
  // download or send blob to your backend
}
```

If you are following the external-app model from the other guides, keep this export helper in your own app layer just like the event bridge and interaction-extension assembly.

## Notes

- The registry components are optional. They are examples of how to react to editor state, not a required runtime model.
- `qti:content:change` and `qti:selection:change` are convenient app-level events for React state synchronization.
- `qti:editor:change` and `qti:editor:selection` are also dispatched on `view.dom`, which can help when your integration is DOM-first.
- Product-specific plugins should stay in your app, while reusable QTI document behavior should live in the shared interaction packages.
