`qti-attributes-panel` displays and edits the schema attributes for the currently selected QTI node and its ancestors.

## Install

```bash
npx shadcn@latest add https://qti-editor.citolab.nl/r/qti-attributes-panel.json
```

**Registry dependencies (auto-installed):** `qti-choice-attributes-editor`, `qti-text-entry-attributes-editor`, `qti-extended-text-attributes-editor`

## Usage

The panel reads the ProseKit editor from the shared `editorContext` (`@lit/context`). Provide the editor higher up in the DOM tree and the panel wires itself up automatically — no property assignment needed.

```ts
import '@/components/attributes-panel';
import '@/components/editor-context';
import { ContextProvider } from '@lit/context';
import { editorContext } from '@/components/editor-context';

// Provide the editor from a host element (e.g. in its constructor):
new ContextProvider(host, {
  context: editorContext,
  initialValue: editor,
});

// Then just render the element anywhere inside that host:
// <qti-attributes-panel></qti-attributes-panel>
```

The panel manages its own attribute event wiring once it receives the ProseKit editor from context.

## How it works

Once the editor is provided through context, the panel subscribes to selection changes. When the cursor moves into a QTI interaction node, the panel reads the schema-defined attributes for that node and its ancestors and renders the appropriate sub-editor. Changes made in the panel are applied as ProseMirror attribute transactions immediately — there is no save step.

The panel dispatches sub-editors based on the active node type:

| Node type | Sub-editor used |
|---|---|
| `qti-choice-interaction` | `qti-choice-attributes-editor` |
| `qti-text-entry-interaction` | `qti-text-entry-attributes-editor` |
| `qti-extended-text-interaction` | `qti-extended-text-attributes-editor` |
| All others | Generic key/value attribute rows |

The three sub-editors are pulled in automatically as registry dependencies when you install `qti-attributes-panel`.
