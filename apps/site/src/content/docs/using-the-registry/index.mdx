The QTI Editor registry is a [shadcn/ui-compatible](https://ui.shadcn.com) GitHub registry. It gives you ready-made UI components that you can copy into your own codebase and adapt without lock-in or template indirection.

QTI Editor has no dependency on shadcn/ui or its component library. The shadcn registry format is used purely as a delivery mechanism: its CLI is a convenient, well-established way to copy individual source files directly into a project. Each `pnpm dlx shadcn@latest add` command downloads the component source into your codebase, installs any npm dependencies it needs, and leaves you with plain files you own and can modify freely. There is no runtime shadcn dependency and no abstraction layer introduced.

You can view this registry as a UI starter. Build the editor itself from `prosekit` and `@citolab/prose-qti`, then copy the components you need to cover common editor UI patterns.

Registry repo: `citolab/qti-editor`

## Prerequisites

These components are Lit web components styled with Tailwind CSS and DaisyUI. Your project needs:

```bash
pnpm add lit tailwindcss daisyui
```

If your framework is not auto-detected by the shadcn CLI, add some manual config first:

- `components.json`
- a `@/*` path alias in `tsconfig.json`
- a minimal `src/lib/utils.ts`

This is required for Angular projects.

## Installing a component

Use the shadcn CLI with the GitHub item address:

```bash
pnpm dlx shadcn@latest add citolab/qti-editor/qti-attributes-panel
```

This copies the component source into your project and installs its npm dependencies. You can then modify the file freely.

After installation, import the copied files from your own app source tree. The exact path depends on where your registry config writes components, so the examples in this chapter use a simple relative path such as `./components/...`. Adjust it to match your project layout.

## Component catalogue

- [QTI Attributes Panel](/docs/using-the-registry/qti-attributes-panel/): schema-aware attribute editor for the current node selection
- [QTI Interaction Insert Menu](/docs/using-the-registry/qti-interaction-insert-menu/): dropdown for inserting supported QTI interactions

Supporting registry items are pulled in automatically when needed:

- `qti-choice-attributes-editor`
- `qti-text-entry-attributes-editor`
- `qti-extended-text-attributes-editor`

## Notes

- Each component is a Lit custom element, which makes the registry framework-agnostic and usable in any HTML context.
- In framework hosts such as Angular or React, assign object values such as the editor instance or `eventTarget` through refs rather than HTML attributes.
- Angular is not auto-detected by the shadcn CLI. Follow the manual setup described in the [Angular guide](/docs/frameworks/angular/) first, then run `pnpm dlx shadcn@latest add`.
- When the shadcn CLI installs a component, it copies the source files into your app and resolves internal imports through your `components.json` aliases. Make sure those aliases point at the directories you want to own.
- Components with `registryDependencies`, such as `qti-attributes-panel`, will automatically pull in their sub-components in a single CLI command.
- The demo app shell in this repository also contains components such as `qti-composer`, `qti-code-panel`, `qti-items-gutter`, `qti-items-navigator`, `qti-convert-menu`, `lit-editor-toolbar`, and `lit-editor-slash-menu`. Those are app-local examples, not registry items.
- In Angular, import the registry files for side effects and add `CUSTOM_ELEMENTS_SCHEMA` to the host component that renders them.
- The copied registry components are optional. The stable integration surface is the public package API plus your own app-level ProseKit assembly.
- The repository's root `registry.json` is the source of truth for the GitHub registry.
